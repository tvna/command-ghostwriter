# Command ghostwriter PRD

本ドキュメントは Command ghostwriter のプロダクト要求仕様 [PRD] である。プロダクトが何を達成すべきか、機能要求・非機能要求・ドメイン制約を記述する。

## 概要・背景・課題

Command ghostwriter は、IaC をはじめとする自動化ツールの導入が困難なインフラ運用現場において、CLI による形式的な運用作業を効率化するためのソリューションである。

このような現場では、繰り返しの多い CLI 実行コマンドを手作業で記述・編集することが多く、転記ミスや作業負荷の増大が課題となる。本プロダクトは、繰り返しが多くなりがちな CLI 実行コマンドを「設定定義ファイル [csv/yaml/toml]」と「Jinja テンプレートファイル」の 2 つに分割して記述する方式を採る。これにより、設定定義ファイルの変更のみで CLI コマンドを生成できるようにする。

さらに、CLI コマンドの生成に留まらず、Markdown による作業手順書の生成にも対応する。

## ターゲットユーザーと利用フロー

本プロダクトは、運用作業の知見を持つ「テンプレート作成者」と、実際に作業を実行する「利用者」という役割の異なる 2 種類のユーザーを想定する。両者の橋渡しとして「ファイルサーバ」と本プロダクト「Command ghostwriter」が位置づけられる。

利用フローは以下のとおりである。

1. ファイルサーバから、テンプレート作成者へ過去のコマンド履歴または手順書が提供される。
2. テンプレート作成者は、コマンドのうち変数となる部分を特定・抽出する。
3. テンプレート作成者は、設定定義ファイルのフォーマット [toml/yaml/csv] を作成し、ファイルサーバへ配置する。
4. テンプレート作成者は、Jinja テンプレートを作成し、ファイルサーバへ配置する。
5. テンプレート作成者は、利用時のルールを整備する。
6. 利用者は、ファイルサーバから設定定義ファイルのフォーマットと Jinja テンプレートを取得する。
7. 利用者は、シナリオに基づくパラメーターを設定定義ファイルに記入する。
8. 利用者は、設定定義ファイルと Jinja テンプレートを Command ghostwriter にアップロードする。
9. Command ghostwriter は、実行可能な CLI コマンドを利用者へ提供する。

この分業により、運用知識を持つ担当者がテンプレートを整備し、現場の利用者はパラメーターの記入のみで成果物を得られる。

## 価値提案・プロダクトゴール

本プロダクトのゴールは、設定の変更のみで再現性のある CLI コマンドや作業手順書を生成できる状態を実現し、形式的な運用作業の効率化と品質の安定化に寄与することである。

提供する価値は次のとおりである。

- 繰り返しの多い CLI コマンドを設定定義ファイルと Jinja テンプレートに分離することで、設定変更のみで成果物を生成できる。
- CLI コマンドだけでなく Markdown 形式の作業手順書も生成でき、運用ドキュメントの整備を支援する。
- 利便性を重視し、必ず Web アプリとして提供する。これにより、利用者は特別な環境構築を要さずにブラウザから利用できる。
- インターネットが利用できない環境を想定し、同じ挙動のままデスクトップアプリとしても提供できる構成を採る。
- 無料で利用可能なプラットフォームとの親和性を重視し、導入の障壁を下げる。

## 技術選定の制約 [非機能・プラットフォーム要求]

プロダクトの提供形態に関する非機能・プラットフォーム要求は以下のとおりである。

- 利便性を考慮し、必ず Web アプリとして提供できること。
- インターネットが利用できない環境を想定し、同じ挙動のままデスクトップアプリにできること。
- Web アプリとデスクトップアプリを、ひとつのプログラミング言語で構成を定義して提供できること。
- 無料で利用可能なプラットフォームと親和性が高いこと。

フロントエンドに関する要求は以下のとおりである。

- フロントエンドは streamlit を利用する Web アプリであること。
- Python のみのコード知識で保守できるよう、可能な限り独自の JavaScript 実装を作らないこと。
- 画面に表示する文字列は可能な限り各ファイルにハードコードせず、`i18n.py` に集約してからインポートして利用すること。

## 機能要求

本プロダクトは、生成のための機能モードと、動作確認のためのデバッグ機能を提供する。

### CLI コマンド生成モード

設定定義ファイルと Jinja テンプレートから、実行可能な CLI コマンドを生成する。

- 変数は `{{ 変数名 }}` の形式で参照する。
- 制御構文 [for, if 等] は `{% %}` で囲んで記述する。
- 生成されたコマンドは、そのままコピー＆ペーストで実行可能であること。

### Markdown 生成モード

設定定義ファイルと Jinja テンプレートから、Markdown 形式の文書を生成する。

- 標準的な Markdown 記法をサポートする。
- コードブロックの言語指定が可能であること。
- テーブル形式での出力に対応すること。
- 画像の参照に対応すること。

### Visual Debug

テンプレートの解析結果を視覚的に確認できる機能を提供する。

- 変数の展開結果の確認。
- テンプレート構文のエラーチェック。
- 制御構文の実行フローの確認。

### TOML/YAML Debug

設定定義ファイルの解析結果を確認できる機能を提供する。

- パース結果の構造化表示。
- データ型の確認。
- 構文エラーの詳細表示。

## 入力仕様

### 設定定義ファイル

設定定義ファイルは、CSV/YAML/TOML のいずれかの形式で記述できる。これらのファイルには、コマンドやドキュメントの生成に必要な変数を定義する。

YAML 形式の例:

```yaml
servers:
  - hostname: web01
    ip: 192.168.1.101
    role: web
  - hostname: db01
    ip: 192.168.1.102
    role: database
commands:
  check_disk: df -h
  check_memory: free -m
```

TOML 形式の例:

```toml
[servers.web01]
hostname = "web01"
ip = "192.168.1.101"
role = "web"

[servers.db01]
hostname = "db01"
ip = "192.168.1.102"
role = "database"

[commands]
check_disk = "df -h"
check_memory = "free -m"
```

CSV 形式の例:

```csv
hostname,ip,role,command
web01,192.168.1.101,web,df -h
db01,192.168.1.102,database,free -m
```

### Jinja テンプレートファイル

Jinja テンプレートファイルは、設定定義ファイルの変数を参照してコマンドや文書を生成するためのテンプレートである。

コマンド生成用テンプレートの例:

```jinja
{% for server in servers %}
# {{ server.hostname }} ({{ server.role }})
ssh {{ server.ip }} "{{ commands.check_disk }}"
ssh {{ server.ip }} "{{ commands.check_memory }}"
{% endfor %}
```

Markdown 生成用テンプレートの例:

```jinja
# サーバー監視手順書

{% for server in servers %}
## {{ server.hostname }}

### 基本情報
- ホスト名: {{ server.hostname }}
- IP アドレス: {{ server.ip }}
- 役割: {{ server.role }}

### CLIコマンド
{{ commands.check_disk }}
{{ commands.check_memory }}

{% endfor %}
```

### テンプレート化の手順

既存のコマンドや Markdown をテンプレート化する手順は以下のとおりである。

1. 変数部分の特定。変数化前のコマンドから、設定によって変化する箇所を抽出する。
2. 設定定義ファイルの作成。抽出した変数を CSV/YAML/TOML のいずれかで定義する。
3. Jinja テンプレートの作成。設定定義ファイルの変数を参照する形でコマンドや文書のひな形を記述する。
4. 動作確認。Visual Debug モードで変数の展開を確認し、エラーがある場合は設定定義ファイルまたはテンプレートを修正する。

## アーキテクチャ要求

バックエンド構成は MVVM モデルに基づいて設計する。`features/` フォルダ内の Python ファイルは、インポートの利便性を考慮してクラス化を徹底すること。

また、パース後のデータに対するメモリバリデーションは実装しない方針とする [修正ループバグの回避策を検討中であるため]。

各モジュールの責務は以下のとおりである。

### core.py

- ビジネスロジックの中核を担う。
- MVVM モデルにおける ViewModel に該当する。

### transcoder.py

- アップロードされたファイルのデコード、およびビジネスロジックで扱うためのエンコードを引き受ける。

### config_parser.py

- TOML ファイル、YAML ファイル、CSV ファイルを辞書形式に変換する。
- CSV ファイルの変換にあたり、引数として列名を決めるための値を取得する。
- ファイルサイズのバリデーションは `validate_uploaded_file` に分離する。
- エラーは例外ではなく `error_message` に出力する。

### document_render.py

- Jinja テンプレートのレンダリングを担う。
- 引数として受け取る `context` は、TOML をパースした結果を想定する。
- フォーマット処理はビジネスロジックであるため、勝手に修正したり破壊したりしないこと。
- ファイルサイズのバリデーション、およびテンプレートのバリデーションは分離する。
- エラーは例外ではなく `error_message` に出力する。`error_message` は、エラーがない場合は None とすること。

### validate_template.py

- DocumentRender に依存する Jinja テンプレートのバリデーションの全てを引き受ける。

### validate_uploaded_file.py

- アップロードファイルのサイズ検証を引き受ける。

## ドメイン制約・セキュリティ要求

### テンプレート検査 [validate_template.py]

テンプレートインジェクション等の悪用を防ぐため、以下のドメイン制約を課す。

- 禁止タグ: `macro`、`include`、`import`、`extends`。
- 禁止属性: `__class__`、`__base__`、`__subclasses__`、`__mro__`、`request`、`config`、`os`、`sys`、`builtins`、`eval`、`exec`、`getattr`、`setattr`、`delattr`、`globals`、`locals`、`do`。
- 大きなループ範囲の上限値として、100000 を閾値として定義する。
- 再帰的構造を検査する静的解析は禁止し、ランタイム処理で例外等を用いて検査するものとする。
- テンプレートインジェクション対策の検査は、計算量が多い検査のみランタイム処理で例外等を用いて検査すること。

### アップロードファイル検査 [validate_uploaded_file.py]

- BytesIO においては、30MB を上限として型バリデーションを適用する。

### 入力ファイルの制約

- ファイルサイズ: 最大 30MB。
- 文字エンコーディング: UTF-8、Shift_JIS、EUC-JP のみテスト済み。
- 改行コード: LF/CRLF 両対応。

### セキュリティ上の注意

- 一部の特殊文字はエスケープが必要である。
- 生成されたコマンドは、実行前に必ず内容を確認すること。

## 用語の定義

本ドキュメントおよび関連ドキュメントで用いる用語を以下に定義する。

- 製品コード: 将来的にリリースされるビジネスロジックを提供するコードである。
- 実装: 製品コードを記述するコードである。
- 期待値: 製品コードにおける値、挙動、状態変化が仕様どおりであるかをテストコードで検証すること。

## 制限事項・既知の制約

既知の制限事項は以下のとおりである。

- 一部の特殊文字はエスケープが必要である。
- テンプレートにおけるネストされたループは非推奨である。
- 入力ファイルのサイズは最大 30MB である。
- 文字エンコーディングは UTF-8、Shift_JIS、EUC-JP のみテスト済みである。
- 生成されたコマンドは、実行前に必ず内容を確認すること。
