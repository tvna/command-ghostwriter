# Node devDependency cooldown bumps + bun-migration remnant cleanup

- Date: 2026-06-10
- Issue: #385

## Problem

The npm -> bun migration is already largely complete (`bun.lock` present, no
`package-lock.json`, CI uses `oven-sh/setup-bun` pinned 1.3.11 +
`bun install --frozen-lockfile`, builds run `bun run dump` / `bun run dist`,
and `docs/standards/package-management.md` mandates `bun add`). Two things
remain: the Node.js devDependencies are stale, and a few npm-era artifacts
linger in `package.json`.

## Policy: 7-day release cooldown (evidence, not speculation)

The maintainer chose a supply-chain-hardening policy: do not adopt a release
younger than 7 days. With today = 2026-06-10, only versions published on or
before 2026-06-03 are eligible. Per-package eligible targets were computed from
the npm registry `time` field (highest stable, non-prerelease version with
publish date <= 2026-06-03):

## Design (approved)

### A. devDependency bumps
Apply the eligible targets above via `bun add -d <pkg>@<ver>` (caret ranges),
which updates `package.json` and `bun.lock` together. commitizen is left as-is.

### B. bun-migration remnant cleanup
- Remove `husky` and `lint-staged` from devDependencies. They are vestigial:
  there is no `.husky` directory and no lint-staged config file; the only
  references are the two devDependency entries and a non-functional
  `"hooks": {"pre-commit": "lint-staged"}` block. Pre-commit is owned by the
  Python `.pre-commit-config.yaml`.
- Remove the `"hooks"` block from `package.json` (husky v4-style key, read by
  nothing in this repo).
- Add `.bun-version` containing `1.3.11` so local bun matches the CI pin
  (`oven-sh/setup-bun@v2` with `bun-version: 1.3.11`). Single source of truth
  for the local toolchain version.
- Leave the `martinbeentjes/npm-get-version-action` references and their
  egress-allowlist comments untouched: that is the real action name reading
  `package.json`, not stale npm-package-manager usage.

### C. Lockfile
`bun.lock` is regenerated by the `bun add` operations; a final
`bun install --frozen-lockfile` confirms it resolves.

## Verification

Local (this environment):
- `bun install --frozen-lockfile` resolves with no lockfile drift.
- `bunx commitlint` loads the config under v21 (commitlint major bump check),
  e.g. a sample `feat: x` message validates and a malformed one fails.
- `bunx prettier --check` runs (prettier 3.8 + sort-json plugin load).
- `cross-env` (v10) and `rimraf` (v6.1) binaries execute under Node 22.
- No Python files change, so pytest / ruff / mypy are unaffected.

Authoritative (CI on the PR):
- `test-and-build-on-pr.yml` runs the full stlite/electron build across OSes.
- electron 42 and @stlite/desktop 0.101 CANNOT be fully verified locally
  (OS-specific packaging, signing). CI is the source of truth; if it surfaces a
  break in the `stlite.desktop` config schema or the electron-builder config,
  that follow-up is in scope for this PR.
