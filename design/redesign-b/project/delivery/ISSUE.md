# UI 全面刷新: Streamlit マルチタブ UI を 2 ペイン・ライブエディタへ置換（Vercel 移行）

## 背景 / Why

現行の Command Ghostwriter は Streamlit ベースの 5 タブ構成（📝コマンド生成 / 📜設定デバッグ / ⚙️詳細設定 / 💼サンプル集 / 🔀ワークフロー）です。Vercel への移行準備を進める中で、UX とアーキテクチャの両面で看過できない課題が顕在化しました。

### 解決したい課題

1. **データソース形式の暗黙判定バグ（最優先）**
   設定定義ファイル（toml / yaml / csv）が拡張子から暗黙的に判定され、誤判定・未対応拡張子で **動作不能** になります。利用者には原因が伝わりません。
2. **生成 ⇔ デバッグのタブ往復**
   コマンド生成タブと設定デバッグタブが分離しており、結果確認のたびに行き来が必要です。
3. **弱いフィードバック**
   ファイル不足でも生成ボタンが押下でき、警告のみで「何が足りないか」が不明瞭。構文エラーはアップロード・生成まで気づけません。
4. **シンプルさの追求による “殺風景” な体験**
   空状態・ガイド・プレビューが乏しく、初見の利用者に 2 ファイル（設定定義 + Jinja テンプレート）の関係が伝わりません。
5. **思想の不一致**
   Streamlit 時代は「CLI スニペットを投入する」ツールでしたが、実運用では **手順書の中にコマンドを埋め込む** 使い方が主でした。

## 提案 / What

Streamlit マルチタブ UI を、**2 ペイン・ライブエディタ**（VS Code スタイル）へ全面置換します。

- **左ペイン = 入力**：データ定義 / テンプレートをタブ切替。データ側に **TOML / YAML / CSV の明示選択**（暗黙判定バグの構造的解消）。行番号付きシンタックスハイライト + 即時バリデーション。
- **右ペイン = 出力**：**手順書（Markdown）をデフォルト表示**。CLI をコードブロックに収め、前後に人の手順を挟む運用ドキュメントとして生成。`Raw` / `Visual Debug` を併設。編集が即座に反映されるライブ生成。
- **詳細設定はモーダルへ退避**（独立タブを廃止）。
- **空状態（オンボーディング）**：2 ファイルの関係を図解し、サンプル投入 / アップロード / テンプレートライブラリの 3 入口を提供。
- **テンプレートライブラリ**：定型作業（Cisco スイッチポート / YAMAHA ルータ / Linux 初期セットアップ / DNS ゾーン / 障害対応手順書 2 種）を出発点として提供。
- **URL ルーティング**：画面状態を URL に同期し、ブラウザの戻る/進む・リロード復元に対応。

### 解決のマッピング

| 課題 | 対応 |
|---|---|
| 暗黙フォーマット判定バグ | フォーマット明示選択 + 自動判定（auto-detect）+ 即時パースエラー表示（該当行ハイライト） |
| 生成 ⇔ デバッグの往復 | 入力↔出力を常時併置（往復ゼロ） |
| 弱いフィードバック | ライブ・バリデーション、未定義変数の検出、状態連動の表示 |
| 殺風景 | オンボーディング図解 + テンプレートライブラリ + 手順書プレビュー |
| 思想の不一致 | 全テンプレートを「CLI を内包する Markdown 手順書」へ刷新 |

## スコープ / 非スコープ

- **In**: 上記 UI の置換、6 テンプレートの初期セット、ライブ生成エンジン（TOML/YAML/CSV パース + 簡易 Jinja レンダラ）。
- **Out（別 Issue）**: 認証・アカウント、テンプレートの永続化/共有、Jinja 全機能対応、クラウド系テンプレート。

## 受け入れ条件

- [ ] toml/yaml/csv を明示選択でき、誤選択時は該当行に具体的なエラーと修復導線が出る（動作不能にならない）。
- [ ] 左ペイン編集が右ペインの手順書・Raw・Visual Debug に即反映される。
- [ ] 6 テンプレートが各 native フォーマットで手順書を生成できる。
- [ ] 画面状態が URL に同期し、戻る/進む・リロードで復元できる。
- [ ] 1024×620 を下回るビューポートでもページスクロールで全体に到達できる。
